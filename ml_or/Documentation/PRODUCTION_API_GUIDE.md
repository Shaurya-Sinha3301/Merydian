# Production API Guide: Base Itinerary vs Optimized Solutions

## Overview

This document clarifies a critical architectural concept: the difference between `base_itinerary` (input) and `optimized_solution` (output), and how they're used in re-optimization scenarios.

## Key Concepts

### Base Itinerary (Input)
The `base_itinerary` is the **constraint framework** or **skeleton template** created by the AI travel agent. It defines:
- Mandatory shared POIs (SKELETON role)
- Optional/flexible POIs (BRANCH role)
- Day structure and time windows
- Start/end anchors (e.g., hotel)

**Role:** Provides the structural constraints that all optimizations must follow.

**Example:**
```json
{
  "days": [
    {
      "day": 1,
      "pois": [
        {"location_id": "LOC_008", "role": "SKELETON"},  // All families must visit
        {"location_id": "LOC_LUNCH", "role": "SKELETON"},
        {"location_id": "LOC_001", "role": "BRANCH"}     // Optional based on preferences
      ]
    }
  ]
}
```

### Optimized Solution (Output)
The `optimized_solution` is the **actual itinerary** generated by the optimizer based on:
- Base itinerary constraints
- Family preferences (must-visit, never-visit, interests)
- Locations and transport data
- Optimization objectives (satisfaction vs cost)

**Role:** The concrete plan with specific POIs, timings, and transport modes for each family.

## Common Misconception

❌ **WRONG:** "When re-optimizing, we pass the previous optimized solution as the base itinerary"

✅ **CORRECT:** "When re-optimizing, we pass the SAME base itinerary with UPDATED family preferences"

## Re-optimization Workflow

### Initial Trip Planning
```python
# AI agent creates base itinerary (skeleton)
base_skeleton = {
    "days": [...],  # Agent's structural template
}

# Optimize with initial preferences
optimizer = ItineraryOptimizer(
    base_itinerary_data=base_skeleton,
    family_prefs_data=initial_preferences,
    locations_data=backend_locations,
    transport_data=backend_transport
)

optimized_v1 = optimizer.optimize_trip(
    family_ids=["FAM_A", "FAM_B", "FAM_C"],
    num_days=3
)
```

### Mid-Trip Preference Change
```python
# FAM_B now wants to visit LOC_006
updated_preferences = modify_preferences(
    family_id="FAM_B",
    must_visit_locations=["LOC_006"]
)

# Re-optimize with SAME base, NEW preferences
optimizer_v2 = ItineraryOptimizer(
    base_itinerary_data=base_skeleton,      # SAME skeleton
    family_prefs_data=updated_preferences,  # UPDATED
    locations_data=backend_locations,
    transport_data=backend_transport
)

optimized_v2 = optimizer_v2.optimize_trip(
    family_ids=["FAM_A", "FAM_B", "FAM_C"],
    num_days=3
)
```

### Generate Cost Delta Explanations
```python
from ml_or.explainability.diff_engine import ItineraryDiffEngine

# Compare TWO optimized solutions
diff_engine = ItineraryDiffEngine()
diffs = diff_engine.compare_optimized_solutions(
    baseline_optimized=optimized_v1,  # Previous plan
    new_optimized=optimized_v2        # New plan
)

# Diffs show: What changed between old and new plans?
# - POIs added/removed
# - Cost increases per family
# - Satisfaction gains/losses
```

## Why This Design?

### 1. Separation of Concerns
- **Base itinerary** = AI agent's structural constraints (changes rarely)
- **Family preferences** = User inputs (change frequently)
- **Optimized solution** = Output result (regenerated each time)

### 2. Consistency
Using the same base itinerary ensures:
- Same skeleton POIs across all versions
- Consistent day structure
- Fair comparison between solutions

### 3. Flexibility
Families can change preferences mid-trip without:
- Violating agent's core plan
- Affecting other families unnecessarily
- Breaking shared constraints (meals, hotels)

## Backend Integration Example

```python
from ml_or.itinerary_optimizer import ItineraryOptimizer
from ml_or.explainability.diff_engine import ItineraryDiffEngine

class TripOptimizerService:
    def __init__(self, database):
        self.db = database
    
    def create_initial_plan(self, trip_id, family_ids):
        """Initial trip planning"""
        # Fetch from backend
        base_skeleton = self.db.get_agent_skeleton(trip_id)
        preferences = self.db.get_family_preferences(family_ids)
        locations = self.db.get_locations(trip_id)
        transport = self.db.get_transport_graph(trip_id)
        
        # Optimize
        optimizer = ItineraryOptimizer(
            base_itinerary_data=base_skeleton,
            family_prefs_data=preferences,
            locations_data=locations,
            transport_data=transport
        )
        
        solution = optimizer.optimize_trip(family_ids, num_days=3)
        
        # Save to database
        self.db.save_optimized_solution(trip_id, version=1, solution=solution)
        return solution
    
    def handle_preference_change(self, trip_id, family_id, changes):
        """Re-optimize after preference change"""
        # Fetch SAME base skeleton
        base_skeleton = self.db.get_agent_skeleton(trip_id)
        
        # Update preferences
        preferences = self.db.get_family_preferences([...])
        preferences = self.apply_changes(preferences, family_id, changes)
        
        # Get latest solution for comparison
        previous_solution = self.db.get_latest_solution(trip_id)
        
        # Re-optimize
        optimizer = ItineraryOptimizer(
            base_itinerary_data=base_skeleton,  # Same!
            family_prefs_data=preferences,      # Updated!
            locations_data=self.db.get_locations(trip_id),
            transport_data=self.db.get_transport_graph(trip_id)
        )
        
        new_solution = optimizer.optimize_trip([...], num_days=3)
        
        # Calculate cost impact
        diff_engine = ItineraryDiffEngine()
        diffs = diff_engine.compare_optimized_solutions(
            baseline_optimized=previous_solution,
            new_optimized=new_solution
        )
        
        return {
            "new_solution": new_solution,
            "cost_impact": diffs,
            "requires_approval": self.check_cost_threshold(diffs)
        }
```

## Summary

| Concept | Type | Changes? | Purpose |
|---------|------|----------|---------|
| **Base Itinerary** | Input | Rarely | Structural constraints |
| **Family Preferences** | Input | Frequently | User desires |
| **Optimized Solution** | Output | Every optimization | Concrete plan |

**Key Takeaway:** The optimizer always starts from the `base_itinerary` skeleton, not the previous optimized solution. To show changes, compare two optimized solutions using the diff engine.
