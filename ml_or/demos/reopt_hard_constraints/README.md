# Re-Optimization Demo - Agentic Workflow Integration

Production-ready demonstration of itinerary re-optimization using the agentic system.

## Overview

This demo showcases how natural language feedback from families is processed through an intelligent agentic workflow to dynamically re-optimize travel itineraries.

### Architecture

```
Chat Message → FeedbackAgent → OptimizerAgent → ExplainabilityAgent → Response
                 (Groq LLM)      (ItineraryOptimizer)  (Gemini LLM)
```

**Key Features:**
- **Session-Based State**: Compatible with both file storage (demo) and database (production)
- **Stateless Processing**: Same logic used for demo and production API endpoints
- **Cumulative Preferences**: All constraints persist across iterations
- **Natural Language**: Processes human feedback, not hardcoded JSON files

## Quick Start

```bash
# Run the demo
python ml_or/demos/reopt_hard_constraints/run_demo.py
```

## What Happens

The demo simulates a chat conversation:

1. **Phase 0**: Initialize trip session with 3 families
2. **Phase 1**: Generate baseline itinerary
3. **Phase 2**: Family A sends: *"We loved Akshardham, we definitely want to visit it tomorrow."*
   - FeedbackAgent extracts: `MUST_VISIT_ADDED`, `poi_id=LOC_006`
   - Optimizer re-runs with updated constraints
   - ExplainabilityAgent generates explanation
4. **Phase 3**: Family B sends: *"We're not interested in Akshardham, please skip it."*
   - FeedbackAgent extracts: `NEVER_VISIT_ADDED`, `poi_id=LOC_006`
   - Preferences are cumulative (FAM_A's must-visit still persists)
   - New itinerary generated with explanations

## Architecture Components

### 1. TripSessionManager (`trip_session_manager.py`)

Manages trip state across feedback iterations:
- Creates and loads sessions
- Stores cumulative preferences
- Tracks itinerary history
- Records feedback audit trail

**Production Migration**: Subclass and override `_save_session` and `get_session` to use database.

### 2. FeedbackProcessor (`feedback_processor.py`)

Stateless processor for handling feedback:
- Loads session state
- Calls AgentController pipeline
- Updates session with results
- Saves outputs

**Production Usage**: Exposed via REST API or WebSocket endpoints.

### 3. Demo Orchestrator (`run_demo.py`)

Simulates website chat interface:
- Initializes trip sessions
- Sends simulated chat messages
- Displays processing results

## Backend Integration

### REST API Example

```python
from fastapi import FastAPI
from ml_or.demos.reopt_hard_constraints.feedback_processor import FeedbackProcessor
from ml_or.demos.reopt_hard_constraints.trip_session_manager import DatabaseSessionManager

app = FastAPI()
processor = FeedbackProcessor()
session_manager = DatabaseSessionManager(db=mongo.db)

@app.post("/api/trips/{trip_id}/feedback")
async def handle_feedback(trip_id: str, family_id: str, message: str):
    result = processor.process_feedback(
        trip_id=trip_id,
        family_id=family_id,
        message=message,
        session_manager=session_manager,
        output_dir=Path("/storage/outputs")
    )
    return result
```

### WebSocket Example

```python
@app.websocket("/ws/trips/{trip_id}/chat")
async def chat_websocket(websocket: WebSocket, trip_id: str):
    await websocket.accept()
    
    while True:
        data = await websocket.receive_json()
        
        # Same processing logic
        result = processor.process_feedback(
            trip_id=trip_id,
            family_id=data['family_id'],
            message=data['message'],
            session_manager=session_manager,
            output_dir=Path("/storage/outputs")
        )
        
        await websocket.send_json(result)
```

## Key Design Decisions

### Why Session-Based?

- **Multi-User Support**: Multiple trips can run concurrently on a website
- **State Isolation**: Each trip's state is independent
- **Easy Migration**: File storage → Database requires minimal changes

### Why Stateless Processor?

- **Scalability**: Can run multiple instances without shared state
- **Testability**: Easy to test with different session configurations
- **Reusability**: Same code for demo, REST API, and WebSocket

### Why Cumulative Preferences?

- **Real-World Behavior**: Users expect their previous preferences to persist
- **Conflict Resolution**: Optimizer handles conflicting preferences gracefully
- **Audit Trail**: Full history of preference changes

## Troubleshooting

**Issue**: `ModuleNotFoundError: No module named 'agents'`
- **Solution**: Ensure you're in the project root and agents/ directory exists

**Issue**: No explanations generated
- **Solution**: Check if Gemini API key is set in `.env`

**Issue**: Session not found
- **Solution**: Ensure trip_id is correct and session file exists in `output/sessions/`
